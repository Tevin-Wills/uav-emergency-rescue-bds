# Research Gap Why It Is a Problem Proposed Solution Method Used & Why Chosen What It Proves / Refutes
1 No standardised coordinate encoding for BDS-3 SMC payloads ASCII: 152–208 bits for data needing only 64 bits — a 69% waste.
Full telemetry struct exceeds GSMC 560-bit global limit, forcing fragmentation with no native reassembly in BDS protocol. 		ESP32 three-mode implementation:
MODE 0 ASCII baseline
MODE 1 Binary 64-bit fixed-point
MODE 3 Dynamic Huffman ~48 bits
Python decoders verify each format independently. 		Method Comparative encoding experiment: same coordinate transmitted in all 3 modes; bit count recorded from Serial Monitor.

Why Direct hardware measurement > theoretical estimation; real compression ratio on live BDS-3 hardware. 		Proves: Binary fixed-point reduces BDS-3 coordinate payload by ≥60% vs ASCII — full telemetry feasible inside GSMC 560-bit limit in a single message.

Refutes: Assumption that ASCII is sufficient for BDS-3 SMC under global GSMC coverage.
2 End-to-end latency of BDS-SMC → software → UAV chain is uncharacterised No study covers full chain: ESP32 → BDS module → GEO satellite → GCC → web platform → UAV GCS.
Without latency budget, impossible to confirm BDS-3 SMC is fast enough for moving survivor (drift >0.5 m/s). 		Instrument full chain:
T1 ESP32 firmware send marker
T2 PC via serial_logger.py
T3 Web platform receipt
Run n=30 transmissions; compute mean, std dev, min, max. 		Method Repeated timestamped logging; n=30, 10 s intervals; mean, std dev, min, max for TX and total latency.

Why n=30 satisfies CLT for mean estimation. Timestamps at both ends eliminate clock ambiguity. 		Proves: First empirical end-to-end BDS-3 SMC latency characterisation. Determines if ~1–2 s round-trip supports UAV waypoint updates to a moving survivor.

Refutes: Assumption that BDS-3 total latency is knowable from individual link studies alone.
3 No empirical RDSS success-rate study across rescue environments GEO elevation 38–52° from Hangzhou; forests, urban canyons, and indoors block LOS.
Probability of a successful BDS-3 SMC cycle in obstructed terrain is empirically unknown. 		Structured field test:
4 environments (open sky, light canopy, urban canyon, indoor)
× 20 transmissions each = 80 total
Log success, fail, latency per attempt. Compute success rate with 95% CI. 		Method Controlled repeated-measures field experiment. field_test_logger.py auto-detects [TX#] markers; logs success/fail/latency. 95% CI per environment.

Why 20 samples/environment gives meaningful CIs; 4 environments span real rescue scenarios; repetition isolates environment as the independent variable. 		Proves: Whether environment significantly affects BDS-3 RSMC success rate. First empirical BDS-3 SMC acquisition dataset in terrain-obstructed rescue scenarios.

Refutes: Springer (2023) gap — RDSS uplink success rate in obstructed terrain was unknown and unstudied.
4 No comparative framework for telemetry encoding schemes across BDS-3 SMC payloads UAV rescue telemetry requires 7 fields (lat, lon, alt, battery, mode, flags, timestamp) in one message.
No published work compares ASCII, binary, Huffman, and AES for this payload — no empirical encoding selection guidance exists. 		telemetry_compare.py: all 4 formats for 7-field struct.
Measure bits, compression ratio vs ASCII, GSMC single-message compatibility.
Output: gap6_telemetry.csv 		Method Multi-format empirical comparison: same 7-field struct in ASCII / Binary / Huffman / AES-128. Bit counts measured; GSMC 560-bit fit assessed in Python.

Why Identical data eliminates confounds. Python mirrors ESP32 firmware — directly comparable to hardware measurements. 		Proves: Which encoding scheme best balances data density, GSMC compatibility, and implementation complexity for full UAV rescue telemetry.

Refutes: Implicit assumption that ASCII is adequate for multi-field BDS-3 SMC telemetry under global GSMC constraints.
